> 本文件依 **工作角色**（您平常在流程裡擔任的職責）整理步驟，方便快速找到自己該看的畫面與按鈕。

## 重要說明：角色與系統權限

| 項目 | 說明 |
|------|------|
| **本手冊的「角色」** | 指 **業務上的分工**（誰負責報修、誰管案件、誰是廠商），方便閱讀；**不是**畫面上的一個下拉選單。 |
| **實際能不能操作** | 由 **後端／SSO 權限** 與 **案件當下狀態**（例如是否輪到廠商節點）決定。同一個帳號在不同案件上可能看到的按鈕不同。 |
| **前端行為** | 登入後 HWMA 選單對所有能進模組的人大致相同；**不會**依角色自動隱藏整個選單（除非路由或全域權限另有設計）。若按鈕灰隱或 API 拒絕，請洽管理員或 IT。 |

## 一、一般使用者（User）

**典型情境**：同仁電腦／螢幕故障，需要 **建立母單（報修案例）** 或 **查看與自己相關的案件／子單**（實際能否看到他人案件依後端資料與權限）。

### 1.1 您最常會用到的功能

| 目的 | 建議操作 | 路徑或入口 |
|------|----------|------------|
| **我要報修（開新母單）** | 進入報修案例列表頁 → 點 **「新增報修」** → 填寫表單並送出 | `/hardware-maintenance` → 新增報修 Modal |
| **用 Case Center／ITCMS 少打字** | 在新增視窗輸入 **issued_no** 後按「get case center data」；或輸入 **device_name** 後按「get ITCMS device-info」 | 同上（Modal 內） |
| **必填別漏** | 務必選擇 **service_type**（PC / Parts / Monitor） | 同上 |
| **我想看某張母單底下有哪些維修子單** | 在列表該列點 **「子單管理」**（需先有列表權限與資料） | 導向 `?caseid=母單編號` |
| **我要進某一筆維修單的詳細流程** | 從子單列表點 **「查看詳情」** | `/hardware-maintenance/<RID>` |

### 1.2 一般使用者建議留意

- **列表預設不會自動載入**：需設定 **issued_status／日期** 等後按 **「搜尋」**（見主手冊 §3）。
- **快速搜尋** 只篩選 **目前這一頁**，換頁後可能要重新輸入關鍵字。
- 送出新增報修成功後，畫面會提示 **issued_no**、**hrt_id**，請自行記錄以便追蹤。

### 1.3 通常「不」由一般使用者操作（除非兼任）

- **價格手冊**（`/hardware-maintenance/pricebook-mgt`）：多由 IT／管理單位維護品項與合約價。
- **廠商判定／SUBMIT 報價**：由案件流程進入廠商節點時，由 **廠商端帳號** 操作（見第三章）。

## 二、IT／服務台／維運

**典型情境**：協助使用者開單、查詢與追蹤 **所有（或部門內）母單與子單**、必要時維護 **可維修品項／合約價**，並協助排除登入或 API 錯誤。

### 2.1 案例列表與查詢（母單層級）

| 目的 | 操作摘要 |
|------|----------|
| 依狀態／日期撈資料 | 勾選 **issued_status**（含字面 `null`）、**start_date**／**end_date**，選 **page_size**，按 **搜尋**。 |
| 在本頁快速找關鍵字 | 使用 **快速搜尋**（僅目前分頁、不打 API）。 |
| 進入某母單的子單總覽 | 該列 **子單管理** → `?caseid=issued_no`。 |
| 時區 | 列表請求帶 **Asia/Taipei**（與主手冊 §8 一致）。 |

### 2.2 子單與流程（維修執行面）

| 目的 | 操作摘要 |
|------|----------|
| 為同一母單 **新建子單** | 子單管理頁 → **新建子單**（成功後重新載入列表）。 |
| 查看單一子單流程、備註、代領人 | 子單卡片 **查看詳情** → RepairFlow（`/hardware-maintenance/<RID>`）。 |
| 在子單上 **新增備註** | 詳情頁 **新增子單備註**（POST comment）。 |
| 設定 **設備代領人** | **設定代領人**（POST proxy；再次送出會覆寫）。 |

### 2.3 價格手冊（通常限 IT／管理者）

| 目的 | 路徑 |
|------|------|
| 維護 **可維修品項**、**合約價** | `/hardware-maintenance/pricebook-mgt` |

- 使用 **reqpir** 後台 API；若出現 **403／無法載入**，請確認帳號權限與後端 Header／環境設定。
- 品項與合約維護完成後，廠商在 **可維修品項** 選料時才會對應到正確資料（與 **GET …/reqpir-items** 連動，依後端實作）。

### 2.4 IT 協助使用者時的檢查清單（建議）

1. 使用者是否已按 **搜尋** 載入列表？  
2. **快速搜尋** 是否因換頁而「看不到」資料？  
3. 廠商無法 SUBMIT：子單 **current_status** 是否為廠商節點？後端 **available_actions** 是否含 SUBMIT？  
4. 價格手冊無法存取：權限是否為管理面向 API？

## 三、廠商（Vendor）

**典型情境**：案件流程進入 **廠商判定／報價** 節點（程式中常見狀態如 **vendor_issue**，實際狀態代碼以後端為準）。此時需 **選取可維修品項**、填 **保固判定** 與 **廠商判斷說明**，並 **送出 SUBMIT**。

### 3.1 進入方式

1. 取得 **子單編號（RID / detail_ticket_no）** 連結，或由內部人員從子單列表帶您進入同一頁。  
2. 網址形式：**`/hardware-maintenance/<RID>`**。

### 3.2 廠商核心操作（RepairFlow）

| 步驟 | 說明 |
|------|------|
| 1 | 確認頁首 **母單資訊**、**RID**、目前 **狀態** 是否為應由廠商處理的階段。 |
| 2 | 在 **維修流程進度** 中，若目前節點提供 **「建立維修項目」**（或同等按鈕），點擊開啟 **廠商判定維修項目（SUBMIT）** 視窗。 |
| 3 | **搜尋品項**：在卡片上方輸入關鍵字（品名、**類別**、規格、機型等），縮小範圍。 |
| 4 | 從 **可維修品項** 卡片點選 **「+」** 加入 **本次報價項目**（卡片會顯示 **類別 + 品名**）。 |
| 5 | 在報價表中為每筆選擇 **保固判定**、確認 **數量**、必要時填 **備註**。 |
| 6 | 填寫 **廠商判斷項目（repaired_issued_msg）**。 |
| 7 | 至少一筆報價項目後，按 **送出 SUBMIT**；成功後視窗關閉，流程往下一節點（依後端轉移）。 |

### 3.3 若畫面上沒有「建立維修項目」

可能原因包括：**流程尚未走到廠商節點**、**您的帳號在後端不被允許執行該 action**、或 **available_actions** 未包含 SUBMIT。請洽開單單位或 IT，並提供 **RID** 與截圖。

### 3.4 廠商通常不需要操作

- **價格手冊後台**（除非貴司分工由廠商維護 catalog）。  
- **新增母單**（除非廠商兼任開單窗口）。

## 四、依情境的快速路徑

以下用「問題」導向，三種角色都可查。

| 我想… | 去哪裡 |
|------|--------|
| 開一張新的報修母單 | `/hardware-maintenance` → **新增報修** |
| 看某母單底下所有子單 | 列表 **子單管理** 或網址 `?caseid=` |
| 看某一筆子單詳情與流程 | `/hardware-maintenance/<RID>` |
| 廠商報價與選料 | 同上詳情頁 → **建立維修項目** → SUBMIT 視窗 |
| 填到料日／後續 transition | 詳情頁依 **可用動作** 按鈕（依狀態顯示） |
| 維護品項／合約價 | `/hardware-maintenance/pricebook-mgt` |
